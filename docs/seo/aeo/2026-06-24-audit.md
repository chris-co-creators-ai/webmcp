## 0. The hard truth up front

`webmcp.md` is a **commercial site selling a $49 "Agent Skill"** that rides on the WebMCP standard. It is **not** the canonical source for "what is WebMCP" — that's the [W3C explainer](https://github.com/webmachinelearning/webmcp) and [Chrome docs](https://developer.chrome.com/docs/ai/webmcp). When an answer engine is asked "what is WebMCP", it will preferentially cite those plus high-authority explainers (VentureBeat, LogRocket, Zuplo, Webfuse, Platinum.ai).

So the AEO goal here is **not** "outrank W3C/Chrome." It is realistically:
1. Get cited as a **secondary/supporting** reference on definitional and "how-to" queries.
2. **Own** the long-tail queries the official docs don't answer well: "agent skill for WebMCP", "how does an agent discover WebMCP tools", "WebMCP llms.txt example", "is my site WebMCP-ready", "WebMCP checkout/booking flow for agents".
3. Be the site that, because it **practices what it preaches** (live `/.well-known/web-mcp`, `/api/mcp`, `/llms.txt`), gets cited as the **reference implementation / live demo**.

Everything below is scored against those realistic goals.

## 2. The 5 citation signals — scored

| Signal | Status | Evidence |
|---|---|---|
| **Direct-answer opener** (definition in first sentence/para) | ⚠️ Weak | Hero leads with the slogan "Operate the web directly." `WhatIsWebMCP` opens with *"The new SEO — but for AI agents"* — a marketing frame, not a citable definition. An LLM extracting "what is WebMCP" finds no clean `WebMCP is …` sentence near an H2 that asks the question. |
| **Questions as H2** | ⚠️ Partial | `WhatIsWebMCP` H2 is *"What is WebMCP, and why now?"* (good — a real question). But `/skill` H1 is *"How to operate a WebMCP site"* (good). Most other headings are slogans ("Without it, the agent guesses", "One pattern, every sector"). No H2/H3 set that mirrors real search queries (vs llms.txt, vs MCP, pricing, is-my-site-ready). |
| **Original data** | ❌ Missing | No proprietary numbers, benchmarks, or comparison tables. The strongest *latent* asset — webmcp.md runs a **live, working WebMCP origin** (4 real tools, real `/api/mcp`) — is never quantified or framed as data ("we expose N tools; here is a real discovery+call transcript with latency"). Competitors win partly on "I made my site agent-ready, here's how" first-person data. |
| **FAQPage schema** | ❌ Missing | No JSON-LD anywhere on the site (`grep` for `ld+json` / `FAQPage` / `@context` → 0 hits). No visible FAQ section either. This is the single biggest, lowest-effort gap. |
| **Recent update-date / freshness** | ❌ Missing | No visible "Last updated" date, no `dateModified`, no `Article`/`TechArticle` schema. The footer shows only `© {year}`. Answer engines weight freshness heavily for a fast-moving 2026 standard; the site emits no freshness signal at all. |

**Score: ~1.5 / 5 signals present.** Machine surfaces are excellent (next section), but the *human-readable, LLM-extractable* layer is under-built for citation.

## 3. Machine surfaces (the site's real strength)

This is where webmcp.md is genuinely differentiated and ahead of every competitor blog.

| Surface | Status | Notes |
|---|---|---|
| `/llms.txt` | ✅ Strong | Clean, agent-addressed, lists tools + protocol + product. One of the best llms.txt files in the niche. |
| `/.well-known/web-mcp` | ✅ Strong | Real discovery doc, 4 tools with JSON Schemas, `force-static`. Genuinely "practices what it preaches." |
| `/api/mcp` | ✅ Present | Callable tool endpoint (GET mirrors discovery, POST executes). |
| `robots.ts` | ✅ Good | Allows `/`, disallows `/download` + `/api/`. **Note:** does not explicitly welcome AI crawlers (GPTBot, ClaudeBot, PerplexityBot, Google-Extended) — currently relies on `*`. Fine, but an explicit allow-list is a cheap trust signal and lets you *prove* you want to be cited. |
| `sitemap.ts` | ✅ Good | 3 URLs, sensible priorities. **But** uses static `changeFrequency`, no `lastModified` dates → another missed freshness signal. |
| Canonical + OG + Twitter | ✅ Good | `layout.tsx` sets canonical, OG, Twitter, keywords, metadataBase. Solid baseline. |

**Caveat / honesty check:** The machine surfaces help **agent actuation and discovery**, and they help the "reference implementation" citation angle. They do **not**, by themselves, get the page cited in a text answer to "what is WebMCP" — an LLM answering that question reads prose, not your tool manifest. Don't conflate "great agent surfaces" with "AEO citations." They are different funnels (see the SEO-lever doc).

## 4. Page-by-page AEO notes

- **`/` (home)** — Marketing-first. The `WhatIsWebMCP` block is the only citation-grade asset and it's diluted by the "new SEO" opener. No FAQ, no schema, no freshness, no comparison table. Highest-leverage page to fix.
- **`/skill` (agent docs)** — Actually the most citable page on the site: real question H1, numbered protocol, tool list, code. Add `TechArticle` + `HowTo` schema and a `dateModified` and this becomes a citation magnet for "how does an agent call WebMCP tools." Currently un-schema'd.
- **`/about`** — Placeholder text ("This is a placeholder portfolio block"). Either fill with a real author bio (E-E-A-T / authority signal — answer engines weight identifiable expertise) or `noindex` it. Shipping a literal placeholder hurts perceived authority.
- **No `/blog` or comparison pages** — The entire "vs" and "how-to" cluster (where competitors win) has no landing surface on webmcp.md. This is the biggest content gap.

## 7. Bottom line

- **Machine surfaces: A.** Best-in-niche llms.txt + /.well-known/web-mcp + /api/mcp. Keep.
- **Citation-worthiness of prose: D.** No direct-answer opener, no FAQ, no schema, no freshness, no comparison pages. Zero current citations.
- **The fix is cheap and known:** the 5 quick wins (direct-answer + FAQ schema + TechArticle/HowTo + freshness + question-H2s) plus 2-3 comparison pages. Effort is low because the content already exists in `site.ts` — it just needs reformatting into citable shapes and wrapping in schema.
