# Technical SEO-audit — WebMCP.md — 2026-06-24

Scope: static analysis of the Next.js 16 App Router repo at `/Users/christianbleeker/Desktop/webmcp`.
No live crawl / GSC / Vercel CWV data was available — CWV findings are code-level risk
assessments, not field data. Factual accuracy checked against the official spec mirror in
`docs/webmcp-spec/`. ANALYSIS ONLY — no code was edited.

## 0. Domain & canonical baseline (read this first)

- Canonical host across the codebase is **`https://www.webmcp.md`** (with `www`), set consistently in
  `src/app/layout.tsx:18`, `src/app/sitemap.ts:3`, `src/app/robots.ts:3`. Internally consistent — good.
- **No apex→www (or www→apex) redirect exists.** `next.config.ts` has no `redirects()` and no
  `trailingSlash` setting. If `webmcp.md` (apex) resolves without a 301 to `www`, the site is
  reachable on two hosts → duplicate-content / split-signals risk. **Decide one canonical host and
  enforce a 301 at the edge (Vercel domain redirect) or in `next.config.ts`.** Severity: **medium**.
- The brand uses a `.md` ccTLD (Moldova) as a pun on Markdown. Google treats `.md` as a
  country-code TLD by default; if the audience is global, set the international target in GSC or use a
  gTLD. Severity: **low** (note for Chris, not a code fix).

## 2. robots.txt

`src/app/robots.ts`:
```
User-agent: *
Allow: /
Disallow: /download, /api/
Sitemap: https://www.webmcp.md/sitemap.xml
Host: https://www.webmcp.md
```

Findings:

| Issue | Detail | Severity |
|---|---|---|
| **`Disallow: /api/` blocks the WebMCP endpoint** | `/api/mcp` and `/.well-known/web-mcp` (the latter is NOT under `/api/`, so it stays crawlable — good) — but `/api/mcp` IS blocked. The whole product thesis is "agents call `/api/mcp`". Robots.txt only governs crawlers, not runtime agent fetches, so functionally agents still work; **but** it signals to AI crawlers (GPTBot, Google-Extended, ClaudeBot) "do not read this surface," which undercuts discoverability of the live tool manifest at `GET /api/mcp`. Consider allowing `/api/mcp` specifically while keeping `/api/checkout` and `/api/download` disallowed. | **medium** |
| **`Disallow: /download` is redundant but harmless** | The page already sets `robots:{index:false}` (`src/app/download/page.tsx:11`). Belt-and-suspenders; fine. The `/api/download` and `/api/checkout` routes are correctly covered by `/api/`. | info |
| **No AI-crawler-specific rules** | For an agent-first site, explicitly *allowing* `GPTBot`, `Google-Extended`, `ClaudeBot`, `PerplexityBot` (or at minimum not blocking them) is on-brand and worth an explicit allow block. | low |
| `Host` directive | Non-standard / Yandex-only; ignored by Google. Harmless. | info |

No wildcard `noindex`, no accidental full-site block. Sitemap reference is correct.

## 3. Metadata infrastructure

`src/app/layout.tsx` is solid:
- `metadataBase` set (`:21`) → relative OG/canonical URLs resolve correctly. Good.
- Title `template: "%s · WebMCP.md"` (`:24`) → child routes get branded titles. Good.
- `alternates.canonical` set to `SITE_URL` at the root (`:52`). **Caveat: this hardcodes the root
  canonical onto the layout.** Because child pages (`/skill`, `/about`) don't override
  `alternates.canonical` in their own `metadata`, Next merges and they may inherit a canonical that
  still points at the page's own URL (Next resolves canonical relative to the route) — verify in
  rendered HTML that `/skill` emits `<link rel="canonical" href=".../skill">` not `.../`. If it
  emits the root URL, `/skill` and `/about` would be canonicalised away → **deindexing risk**.
  Severity: **medium — needs render verification.**
- OG/Twitter complete; `opengraph-image.jpg` exists at `src/app/opengraph-image.jpg` (105 KB) and is
  auto-wired by App Router. Twitter card `summary_large_image`. Good.

Per-route `generateMetadata` / static `metadata`:

| Route | Title | Description | Canonical override | Gap |
|---|---|---|---|---|
| `/` | from layout default | yes | inherits | OK |
| `/skill` (`skill/page.tsx:8`) | "Agent documentation" | yes | **none** | verify canonical (see above) |
| `/about` (`about/page.tsx:7`) | "About" | yes (thin: "About the person behind WebMCP.md.") | **none** | weak description; thin page (§6) |
| `/download` (`download/page.tsx:8`) | "Download" | **none** | n/a | `index:false` set — correct, description absent is fine |

- **No JSON-LD / structured data anywhere.** For a product+docs site, `SoftwareApplication` /
  `Product` + `Offer` (the $49 Skill) and `Organization` schema are cheap wins for rich results and
  for AI-crawler comprehension. Severity: **medium** (recommended add, no schema today).
- No `manifest.webmanifest` / theme-color. PWA-level, low priority. `icon.svg` + `apple-icon.tsx`
  cover favicons. Severity: **low**.

## 4. Canonicalization strategy

- Single canonical host (`www.webmcp.md`) — good, **once the apex→www 301 exists** (§0).
- Root canonical is hardcoded in layout (§3) — the one real risk: child routes must emit their own
  self-referential canonical, not the root. **Action: confirm in built HTML; if wrong, set
  `alternates.canonical` per route or switch the layout to not hardcode and let each page self-canonicalise.**
- Hash-anchor URLs (`/#pricing`, `/#what`, `/#agents`, `/#self-demo`) are used heavily for nav and CTAs.
  Fragments are not separate URLs to Google — no duplicate-content issue, but it means **all primary
  content lives on one indexable URL (`/`)**. That concentrates ranking on the homepage; fine for a
  one-pager, but limits long-tail capture. Consider whether `/skill` should absorb more of the
  agent-protocol content as its own indexable surface.

## 7. Agent / machine discoverability (the site's whole point)

| Surface | File | Status | Finding |
|---|---|---|---|
| `/llms.txt` | `public/llms.txt` | served static | Well-formed, lists tools + protocol + endpoints. Good. Add `lastModified`/date and the canonical `https://www.webmcp.md` absolute base inside it (currently uses root-relative paths only — fine for same-origin agents, but absolute URLs help off-origin LLM crawlers). Low. |
| `/.well-known/web-mcp` | `.well-known/web-mcp/route.ts` | `force-static`, JSON manifest | Good — crawlable (not under `/api/`). Mirrors `/api/mcp` GET. |
| `GET /api/mcp` | `api/mcp/route.ts` | JSON manifest | **Blocked by `Disallow: /api/`** in robots (§2). Functionally callable, but AI crawlers told to skip it. |
| `POST /api/mcp` | `api/mcp/route.ts` | tool executor | Correct shape (`{tool,args}` → `{ok,...}`). Safety: `start_checkout` returns approval URL, never charges — matches spec's "sensitive actions require confirmation." Good. |

- **Manifest duplication risk:** the tool manifest is hand-copied in TWO places —
  `.well-known/web-mcp/route.ts` (inline literal) and `api/mcp/route.ts` (`MANIFEST` const). They are
  currently identical but will drift. **Recommend a single source** (e.g. export `MANIFEST` from
  `src/lib/site.ts` and import in both). Not strictly SEO, but a discoverability-integrity issue:
  if the two manifests disagree, agents get inconsistent tool lists. Severity: **low/medium**.
- No `Content-Type` issue: routes use `NextResponse.json` → `application/json`. Good.

  ("mirrors the .nl marquee" in `src/lib/site.ts:18`; "Back to webmcp.md"). No `app/[locale]`,

- **Recommendation when an EN/NL split ships:** introduce `app/[locale]/...`, set

  per-locale entries to `sitemap.ts`. Until then, **do not add hreflang** (pointing at a
  non-existent `.nl` would be an error). Severity: **n/a today / planning item.**

## 10. Factual accuracy — site copy vs. official spec (overclaim audit)

The official mirror (`docs/webmcp-spec/INDEX.md:7-11`, `chrome/overview.md`) states WebMCP is a
**proposed standard**, a **Draft Community Group Report (2026-03-09)**, **under incubation, not yet a
W3C Standard**, available only as a **Chrome 149 origin trial (experimental)** + a local dev flag.
Several site strings overclaim — this is a **trust/E-E-A-T and accuracy risk**, and for an
agent-facing site, factual drift directly undermines credibility:

| Site string | Where | Spec reality | Verdict |
|---|---|---|---|
| "Chrome 149 stable" | `site.ts:20` (TICKER), `site.ts:100` (TIMELINE "Chrome 149 / Stable") | Chrome 149 **origin trial**, experimental, behind a flag | **Overclaim — false.** Change to "Chrome 149 origin trial". |
| "Gemini in Chrome: shipping" / "Agentic browsing shipped to the stable channel in 2026" | `site.ts:21`, `site.ts:92` (CONCEPT_CARDS) | Not supported by the mirror; WebMCP is experimental | **Overclaim — unverified.** Soften or cite. |
| "Browser-native agents reach the stable channel" | `site.ts:100` | Origin trial only | **Overclaim.** |
| "Announced at Google I/O 2026 (May 19)" | `site.ts:20`, `site.ts:99` | Chrome doc published **May 18, 2026**; I/O framing is plausible but the "stable" framing around it is not | Date OK; surrounding "stable" claims not |
| "WebMCP = proposed open web standard (W3C)" | `site.ts:24`, Footer `Sections.tsx:250-251` ("proposed open web standard. Not affiliated with Google or Microsoft.") | **Matches spec.** | **Correct — keep.** Good that the footer already hedges + disclaims affiliation. |
| "open web standard from Google & Microsoft" | `Hero.tsx:30` | Joint Google+Microsoft initiative in W3C WebML CG — accurate as *initiative*, but "standard" should be "proposed standard" | **Minor overclaim** — add "proposed". |

**This is the highest-impact non-technical finding.** The site simultaneously says "proposed
standard" (footer, correct) and "Chrome 149 stable / shipping" (ticker+timeline, false). The
contradiction is visible on the homepage. Recommend a copy pass in `src/lib/site.ts` to align every
status claim to "origin trial / proposed standard / under incubation." Owner: **content/copy**, not
SEO — flagged as a cross-agent dependency. Severity: **medium (accuracy/trust).**
