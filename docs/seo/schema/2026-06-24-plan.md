## How to wire it (App Router pattern)

Render JSON-LD as a plain `<script>` in the Server Component for each route. Do **not** use
`next/script` with `type="application/ld+json"` (next/script is for executable JS; for LD+JSON a
raw `<script>` in the server-rendered HTML is correct and crawlable). Pattern:

```tsx
<script
  type="application/ld+json"
  // eslint-disable-next-line react/no-danger
  dangerouslySetInnerHTML={{ __html: JSON.stringify(jsonLd) }}
/>
```

Recommended: a tiny helper `src/lib/seo/jsonLd.ts` exporting builder functions that read from
`src/lib/site.ts`, so schema stays in sync with the product config. Snippets below are written as
those builders.

## 1. Organization + WebSite (site-wide ‚Üí render in `layout.tsx`)

**Page:** every page (root layout `<body>`, before `{children}`).
**File:** `src/app/layout.tsx`.
**Required fields:** Organization ‚Üí `name`, `url`. WebSite ‚Üí `name`, `url`.
**Why one `@graph`:** emit both nodes in a single `<script>` so they cross-reference by `@id`.

```tsx
// src/lib/seo/jsonLd.ts
import { SITE } from "@/lib/site";

const SITE_URL = "https://www.webmcp.md";

export function orgAndWebsiteJsonLd() {
  return {
    "@context": "https://schema.org",
    "@graph": [
      {
        "@type": "Organization",
        "@id": `${SITE_URL}/#organization`,
        name: SITE.name,
        url: SITE_URL,
        description: SITE.tagline,
        logo: {
          "@type": "ImageObject",
          url: `${SITE_URL}/apple-icon.png`,
        },
      },
      {
        "@type": "WebSite",
        "@id": `${SITE_URL}/#website`,
        name: SITE.name,
        url: SITE_URL,
        description: SITE.tagline,
        publisher: { "@id": `${SITE_URL}/#organization` },
        inLanguage: "en",
      },
    ],
  };
}
```

> `logo` points at `/apple-icon.png` ‚Äî `src/app/apple-icon.tsx` exists and renders a PNG icon.
> If a dedicated wide logo asset is added later, swap the URL. The file is load-bearing for the
> Organization rich result; do not leave it blank.
> No `sameAs` (no verified social profiles found in source). Omit rather than invent ‚Äî adding a
> fabricated LinkedIn/X URL is worse than omitting `sameAs`.

Render in layout:

```tsx
// inside <body>, before {children}
<script
  type="application/ld+json"
  dangerouslySetInnerHTML={{ __html: JSON.stringify(orgAndWebsiteJsonLd()) }}
/>
```

## 2. Product + Offer (the $49 Agent Skill ‚Üí home page)

**Page:** `/` (the Skill is sold on the home `#pricing` section).
**File:** `src/app/page.tsx` (Server Component ‚Äî add the `<script>` directly in the returned tree).
**Impact:** highest ‚Äî this is the merchant-listing / product rich-result win (price shown in SERP).
**Required fields (Product):** `name`, plus `offers`. **Required (Offer):** `price`,
`priceCurrency`, `availability`, and `url` (the buy/landing URL). Google also wants `image` and a
review/rating for the full product snippet ‚Äî see dependency note.

```tsx
// src/lib/seo/jsonLd.ts
import { SKILL } from "@/lib/site";

export function skillProductJsonLd() {
  return {
    "@context": "https://schema.org",
    "@type": "Product",
    "@id": `${SITE_URL}/#agent-skill`,
    name: SKILL.name,
    description: SKILL.summary,
    sku: SKILL.id,                    // "webmcp-agent-skill"
    image: `${SITE_URL}/images/skill-hero.jpg`,
    brand: { "@type": "Brand", name: SITE.name },
    category: "Software > AI Agent Skill",
    softwareVersion: SKILL.version,   // valid Product property
    offers: {
      "@type": "Offer",
      url: `${SITE_URL}/#pricing`,
      price: (SKILL.priceCents / 100).toFixed(2),   // "49.00"
      priceCurrency: SKILL.currency.toUpperCase(),  // "USD"
      availability: "https://schema.org/InStock",
      priceValidUntil: "2026-12-31",
      itemCondition: "https://schema.org/NewCondition",
      seller: { "@id": `${SITE_URL}/#organization` },
    },
  };
}
```

> ALTERNATIVE shape to consider: because the Skill is software, `SoftwareApplication`
> (`applicationCategory: "DeveloperApplication"`, `operatingSystem: "Any"`) is also semantically
> valid and can earn an app rich result. **Recommendation: ship `Product` first** (price rich
> result is the stated goal). Optionally add a second `SoftwareApplication` node in the same
> `@graph` if app-install snippets become a priority. Do **not** stack both as competing primary
> entities without the `@graph` wrapper.

> `priceValidUntil` is a static near-future date to satisfy Google's recommendation; bump it
> annually or compute `new Date().getFullYear() + "-12-31"`.

> `aggregateRating` / `review` intentionally OMITTED ‚Äî no real reviews exist. Adding fake ratings
> is a Google manual-action risk. See dependency D2.

## 3. HowTo (the agent protocol ‚Üí `/skill`) ‚Äî REALIGNED to the real spec

> **Changed in rev. 2.** WebMCP is an **in-browser API**, not a `/.well-known` + REST model. Per
> `docs/webmcp-spec/w3c/README.md` ("Backend Integrations vs. In-browser WebMCP Tools" +
> "Lifecycle of a Tool Call"), tools are registered **on the page** ‚Äî imperatively via
> `document.modelContext.registerTool({ name, description, inputSchema, execute })`
> (`chrome/imperative-api.md`) or declaratively via `<form toolname tooldescription>`
> (`chrome/declarative-api.md`). The HowTo now describes the in-browser flow, not a server fetch.

**Page:** `/skill` (renders `AGENT_STEPS` as a visible ordered list ‚Äî the HowTo prerequisite, a
matching visible procedure, is satisfied).
**File:** `src/app/skill/page.tsx`.
**Required fields:** `name`, `step[]` each a `HowToStep` with `name` + `text`.

**Step source of truth:** the builder maps over `AGENT_STEPS` from `src/lib/site.ts`, so the schema
text byte-matches the on-page list. The four steps are now **LOCKED** to `content`'s canonical copy
in `docs/seo/content/realignment.md` ¬ß1 (received 2026-06-24, byte-for-byte). The builder is
unchanged; only the `AGENT_STEPS` strings change.

> **POV correction from `content` (applied):** the steps are written from the **agent's** point of
> view (operating a site), not the site author's. There is deliberately **NO "register tools"
> step** ‚Äî registration is the developer's setup, not part of the agent's operation loop. The
> HowTo describes the four-step operation loop, matching the rest of the page.

Canonical `AGENT_STEPS` copy (byte-for-byte from `content`; `s.title` = the line after the number,
`s.body` = the paragraph):

1. **Discover registered tools** ‚Äî On the open page, query the browser for the tools it exposes. The
   site registers them in JavaScript with `document.modelContext.registerTool(...)` or by annotating
   an HTML `<form toolname>`. If tools are present, call them ‚Äî stop scraping the DOM.
2. **Read each tool's schema** ‚Äî Every tool carries a name, a plain-language description, and a
   JSON-Schema `inputSchema` for its arguments. Read the schema to learn the exact fields, types,
   and enums before you build a call.
3. **Call the tool ‚Äî the browser runs it visibly** ‚Äî Invoke the tool with structured arguments that
   validate against its `inputSchema`. The browser executes it on the page in front of the user: an
   imperative tool runs its `execute()` callback; a declarative form is brought into focus and
   populated. Never invent fields the schema doesn't list.
4. **Verify & recover from the result** ‚Äî Read the structured result the tool returns. On a
   validation or execution error, repair the arguments from the schema and the error message, then
   retry ‚Äî never fabricate success.

```tsx
// src/lib/seo/jsonLd.ts
import { AGENT_STEPS } from "@/lib/site";

export function agentProtocolHowToJsonLd() {
  return {
    "@context": "https://schema.org",
    "@type": "HowTo",
    name: "How an AI agent operates a WebMCP site",
    description:
      "The in-browser WebMCP protocol: how a browser AI agent discovers the tools a page registers via document.modelContext (or declarative <form> annotations), calls them with schema-valid arguments, and verifies the structured result.",
    totalTime: "PT2M",
    step: AGENT_STEPS.map((s, i) => ({
      "@type": "HowToStep",
      position: i + 1,
      name: s.title,
      text: s.body,
      url: `${SITE_URL}/skill#step-${i + 1}`,
    })),
  };
}
```

> The `url` per step uses `#step-N` anchors. These anchors do **not** exist in the markup today.
> Either (a) add `id="step-1"‚Ä¶` to each `<li>` in `src/app/skill/page.tsx`, or (b) drop the
> per-step `url` field (it's optional). Flagged as dependency D3.

> NOTE: Google deprecated the HowTo *rich result* in 2023, so this won't render stars/steps in
> SERP, but HowTo remains valid structured data that LLM crawlers and agents consume ‚Äî especially
> on-brand for an agent-first site. Keep it; it's low-effort and semantically correct.

## 4. BreadcrumbList (sub-pages ‚Üí `/skill`, `/about`)

**Pages:** `/skill`, `/about` (and any future deep route). **Not** on `/` (home is the root ‚Äî
a single-item breadcrumb is noise).
**Files:** `src/app/skill/page.tsx`, `src/app/about/page.tsx`.
**Required fields:** `itemListElement[]` each with `position`, `name`, `item`.

```tsx
// src/lib/seo/jsonLd.ts
export function breadcrumbJsonLd(trail: { name: string; path: string }[]) {
  return {
    "@context": "https://schema.org",
    "@type": "BreadcrumbList",
    itemListElement: trail.map((c, i) => ({
      "@type": "ListItem",
      position: i + 1,
      name: c.name,
      item: `${SITE_URL}${c.path}`,
    })),
  };
}
```

Usage:

```tsx
// /skill
breadcrumbJsonLd([
  { name: "Home", path: "/" },
  { name: "Agent documentation", path: "/skill" },
]);

// /about
breadcrumbJsonLd([
  { name: "Home", path: "/" },
  { name: "About", path: "/about" },
]);
```

> Status copy in these answers is already correct ("proposed open web standard", "W3C Draft
> Community Group Report", "Chrome 149 origin trial" ‚Äî not "stable"), consistent with the ¬ß3 status
> facts. Builder JSON-stringifies the strings, so the straight apostrophes and em-dashes in the
> answers escape cleanly into valid JSON-LD.

The FAQPage builder takes `content`'s array verbatim (keys `question` / `answer` confirmed):

```tsx
// src/lib/seo/jsonLd.ts
// Caller passes the SAME FAQ array that renders the visible <section> (from src/lib/site.ts):
//   import { FAQ } from "@/lib/site";  faqJsonLd(FAQ)
// content's canonical array (realignment.md ¬ß3) uses { question, answer } ‚Äî keys confirmed 2026-06-24.
export function faqJsonLd(faq: { question: string; answer: string }[]) {
  return {
    "@context": "https://schema.org",
    "@type": "FAQPage",
    mainEntity: faq.map(({ question, answer }) => ({
      "@type": "Question",
      name: question,
      acceptedAnswer: { "@type": "Answer", text: answer },
    })),
  };
}
```

> The `faq` argument MUST be the SAME `src/lib/site.ts` constant that renders the visible FAQ, so
> the schema text byte-matches the on-page text. Do not hand-author divergent answer strings here ‚Äî
> the 9 canonical Q&As live in `content`'s `FAQ` array.
